### 2026-06-27 — New feature: company credit-rating lookup (Credit-Ratings/)
- **Goal:** input a company NAME -> fetch the AI-recommended credit rating (the value Google/Bing show in their AI overview, e.g. "HP S&P Global company rating" -> BBB) -> map it onto the analyst's dropdown scale image (Moody's/S&P pairs: A1/A+, A2/A, A3/A-, Baa2/BBB, Ba1/BB+, Ba2/BB, B1/B+, B2/B, B3/B-, Caa/CCC, Ca2/CC, C2/C, D2/D) -> return JSON {company, rating, ...}.
- **User choices (asked up front):** (1) fetch via **Firecrawl AI overview** scrape of Google results (same Firecrawl/residential-IP approach as beta_route.py); (2) **snap to nearest** dropdown value; (3) accept **either agency** (S&P or Moody's), whichever the snippet gives.
- **Built:** `Credit-Ratings/credit_route.py` — ONE self-contained APIRouter file (folder has a hyphen so it can't be a Python package). Exposes `GET /api/credit-rating?company=`. Wired into `app.py` via `importlib.util.spec_from_file_location` (`_load_router_from_path`) since `import Credit-Ratings...` is invalid.
- **Mapping logic:** `CANON` = full 22-notch S&P/Moody's ladder with ranks; `DROPDOWN` = the 13 image values tagged by S&P notch. `map_rating` snaps a fetched token to the min-rank-distance dropdown entry, tie-break **prefer same S&P letter family** then better rating — so BBB+/BBB/BBB- ALL -> Baa2/BBB, AAA/AA/A+ -> A1/A+ (dropdown caps at A+), BB- -> Ba2/BB. `parse_rating` regex-matches all rating tokens (longest-first so BBB+ beats BBB), scores each by proximity to context words (rating/credit/S&P/Moody/outlook) + length, drops bare single letters with no nearby context. Agency detected case-sensitively (S&P all-caps AAA vs Moody's Aaa). Outlook parsed if stated.
- **Verified LIVE (real Firecrawl key from .env):** HP -> A- -> **A3/A-** (Stable); Microsoft -> AAA -> **A1/A+** (snapped to cap); Ford -> BBB- -> **Baa2/BBB** (Negative). Offline unit-checked all 22 notches + Moody's tokens map correctly. Standalone CLI: `python Credit-Ratings/credit_route.py "HP"` (loads FIRECRAWL_API_KEY from .env itself).
- **Notes:** Firecrawl required (no useful no-key fallback for a search-engine scrape -> 503 if unset). Primary = scrape google.com/search markdown; fallback = Firecrawl `/search` snippets if no rating parsed. In `app.py`'s route dump the 3 routers show as `_IncludedRouter` (this FastAPI version includes lazily) — they ARE mounted; not a bug. Did NOT touch requirements.txt (`requests` already transitively present via yfinance/beta).

### 2026-06-26 — Exchange-prefixed ticker input (e.g. XPAR:CAP) + 4-row cap + 4th-year projection
- **3 related Simply tweaks (all in `simply_route.py`, plus a small `data.py` change):**
  1. **Cap output to 4 periods** (`_MAX_ROWS=4`, applied in `_scrape`) — user's Excel template expects 4 values/series; SWS was returning 5 (e.g. 2025–2029) → "Expected 4 values each, got 5/5". Rows are oldest-first so it keeps latest-actual + next 3 and drops the rest. No SWS math changed.
  2. **Project a 4th year when SWS gives only 3** (`_project_next`/`_fill_to_max`) — formula per metric: `value(next)=value(last)*(value(last)/value(prev))` (same YoY growth as the last published interval), eps→2dp else int. Appends a row tagged `estimated:True` (only in JSON; Excel/grouped read fixed cols so shape unchanged). NEVER alters real values. Needs ≥2 real rows.
  3. **Exchange-prefixed input** (user choice: MAP, not just strip): `XPAR:CAP` → symbol `CAP` + exchange hint `epa`. Pure strip was WRONG — bare "CAP" resolved to Chilean snse-cap, not Capgemini. Built `_EXCHANGE_MAP` (MIC→SWS slug) **verified LIVE via Firecrawl search**: XPAR=epa, XLON=lse, XETR=etr, XAMS=ams, XBRU=ebr, XMIL=bit, XMAD=bme, XSWX=vtx, XSTO=sto, XCSE=cph, XHEL=hel, XLIS=eli, XOSL=ob, XASX=asx, XTSE=tsx, XHKG=hkg, XTKS=tse, XKRX=kose, XNAS=nasdaq, XNYS=nyse. Unknown prefix → raw prefix lowercased (handles users typing SWS-style codes). `_scrape` splits on first ':' (partition), maps prefix, doesn't override an explicit exchange.
- **`data.py` change:** `_parse(html, ticker, exchange=None)` now collects ALL symbol matches and, when an exchange is given, PREFERS the result whose SWS slug starts with `<exchange>-` (e.g. `epa-cap`) instead of taking the first hit — robust against search-ranking variance. Both `find_url` call-sites pass `exchange`.
- **Verified:** XPAR:CAP→Capgemini (epa-cap), XLON:HSBA→HSBC (lse-hsba), AAPL→Apple (unchanged); 4-row cap on NKE/MSFT; projection unit-test 3→4 rows correct. Purged a stale wrong `CAP|` (Chilean) entry from `sws_url_cache.json` left by an earlier test.
- **Gotcha:** SWS slugs ≠ MIC codes (XPAR→epa, XETR→etr, XKRX→kose). Don't assume; the map was built from live probes. `_parse` exchange-preference + the query hint together do the disambiguation.

### 2026-06-01 — Fix Korea (DART) 524 over Cloudflare tunnel
- **Symptom:** Korea fetch showed "failed (524)" in the UI when run through the trycloudflare quick-tunnel. Backend was actually fine — the job ran to completion (573 pages scanned, classification done); only the browser saw an error.
- **Root cause:** `/api/extract-from-korea` called `resolve_kr_company_number()` **synchronously inside the request** before returning the job_id. On a cold start, dart-fss downloads/parses the full DART corp-code list (~90s, measured), which exceeds Cloudflare quick-tunnels' fixed ~100s edge timeout → 524. Other sources return a job_id instantly (poll pattern), so only Korea hit this. (524 is a Cloudflare-only status; neither DART nor FastAPI emits it.)
- **Decided:** Move the resolve into the background pipeline (Approach A). The endpoint now creates the job with raw ticker/company_name and returns the job_id immediately (~0.14s); `run_extraction_pipeline`'s Korea branch resolves the corp_code (renaming the PDF to `{ticker or corp_code}_KR_{category}.pdf`) right before the fetch. Rejected: B) warm DART corp-list cache at startup, C) both — A structurally fixes it and matches the existing job/poll architecture.
- **Verified:** POST now returns in 0.14s; background resolve completed in ~90s (005930→00126380 Samsung); no 524. Files changed: `backend.py` only (endpoint + KR branch of pipeline). Extraction math untouched.
- **Note:** Cloudflare free quick-tunnels have a fixed ~100s origin timeout that can't be raised — any in-request work must stay under it; use the job/poll pattern for slow work.
